## Назначение

Используй этот skill, когда пользователь сообщает информацию, которую нужно сохранить, учесть в будущем, привязать к курсу, дате или превратить в напоминание/дедлайн.

Главное правило: не сохраняй всё в одну память. Сначала выбери правильный тип информации и соответствующий tool.

## Доступные tools

### `remember_user_observation`

Сохраняет долговременное наблюдение о пользователе в analytics/user memory.

Используй, если информация относится к пользователю в целом: устойчивые предпочтения, привычки, стиль обучения, стиль работы, коммуникационные предпочтения, ограничения или долгосрочные интересы.

Примеры:

- "Запомни, я лучше учусь вечером."
- "Мне не нравятся длинные теоретические объяснения."
- "Я предпочитаю короткие практические задачи."
- "Я хочу развиваться в backend-разработке на Python."

Не используй этот tool, если информация относится к конкретному курсу, конкретной дате, напоминанию или дедлайну.

### `create_course_observation`

Сохраняет наблюдение по конкретному курсу.

Используй, если пользователь сообщает предпочтение, ограничение, цель, сложность, прогресс или контекст, связанный с конкретным курсом.

Курс выбирай из системного контекста. Если курс только что создан через `create_course`, используй id этого курса. Не придумывай course_id.

Примеры:

- "По курсу итальянского я не хочу заниматься утром, только вечером по будням."
- "Python я учу для backend-разработки, а не для Data Science."
- "В курсе английского мне важнее speaking, чем грамматика."
- "По математике мне тяжело даются производные."

Если пользователь явно говорит "по курсу", "в курсе", называет существующий курс или говорит о цели обучения внутри курса, предпочитай `create_course_observation`, а не `remember_user_observation`.

### `create_date_observation`

Сохраняет контекст на конкретную будущую дату или период.

Используй, если пользователь сообщает событие, занятость, ограничение или важный контекст, привязанный к дате/периоду.

Примеры:

- "Через два дня я пойду к врачу."
- "Завтра у меня мало времени, максимум час вечером."
- "7 мая я поеду к бабушке."
- "На следующей неделе у меня экзамен."
- "В пятницу вечером я занят."

Это не долговременная память. Это date observation, который потом подгружается как системный контекст и может использоваться workflow/agent logic.

### `read_day_observations`

Читает итоговые day observations за конкретный день.

Используй, если пользователь спрашивает, что он делал сегодня, вчера или в конкретную дату.

Примеры:

- "Что я сегодня делал?"
- "Что я делал вчера?"
- "Покажи записи за 2026-06-15."
- "Есть ли итоги за сегодня?"

Важно: `read_day_observations` только читает уже созданные workflow-записи. Он ничего не создает.

### `create_reminder`

Создает напоминание.

Используй, если пользователь просит напомнить о действии в конкретное время или через относительный интервал.

Примеры:

- "Напомни завтра в 18:00 позвонить преподавателю."
- "Напомни через час проверить деплой."
- "В пятницу утром напомни купить продукты."

Reminder — это точечное уведомление. Не сохраняй такие запросы как user memory или date observation.

### `create_deadline`

Создает дедлайн.

Используй, если пользователь сообщает срок, дату сдачи, экзамен, обязательство или дату, к которой нужно что-то выполнить.

Примеры:

- "До пятницы нужно сдать проект."
- "Экзамен по итальянскому 20 июня."
- "К понедельнику нужно закончить README."
- "Курс по Python нужно закончить до конца месяца."

Deadline — это срок выполнения, а не напоминание. Если пользователь просит именно уведомить в конкретное время, используй `create_reminder`.

## Routing rules

Перед вызовом tool определи тип информации:

- Устойчивая информация о пользователе в целом → `remember_user_observation`.
- Информация о конкретном курсе → `create_course_observation`.
- Событие/занятость/ограничение на дату или период → `create_date_observation`.
- Просьба напомнить в конкретное время → `create_reminder`.
- Срок выполнения или дата сдачи → `create_deadline`.
- Вопрос о том, что было сделано в конкретный день → `read_day_observations`.

## Важные разграничения

### Course observation vs user observation

Если информация касается конкретного курса, используй `create_course_observation`.

Пользователь:
"По курсу итальянского я не хочу заниматься утром, только вечером по будням."

Правильно:
`create_course_observation`

Неправильно:
`remember_user_observation`

### Date observation vs user observation

Если информация привязана к дате или периоду, используй `create_date_observation`.

Пользователь:
"Через два дня я пойду к врачу."

Правильно:
`create_date_observation`

Неправильно:
`remember_user_observation`

### Day observation

Interactive agent не создает day observations напрямую.

Если пользователь сообщает, что он сегодня делал, изучал, не успел или выполнил, не вызывай tool для сохранения day observation. Session-close workflow позже прочитает transcript и сам создаст итоговое day observation.

Примеры:

- "Я сегодня учил Python."
- "Сегодня разобрал FastAPI."
- "SQL сегодня не успел."

Правильное поведение: ответить поддерживающе и помочь продолжить работу, но не сохранять это через `remember_user_observation` или `create_date_observation`.

## Как формулировать observations

Формулируй observation коротко, от третьего лица, без слов "пользователь сказал, что".

Хорошо:

- "Пользователь предпочитает короткие практические задания вечером."
- "Пользователь не хочет заниматься итальянским по утрам; предпочитает вечера по будням."
- "Пользователь будет занят через два дня из-за визита к врачу."

Плохо:

- "Я люблю короткие задачи."
- "Запомнить: вечером."
- "Пользователь сказал, что он не хочет..."

## Ответ пользователю после tool call

После успешного сохранения скажи коротко:

- что сохранено;
- где это будет учитываться.

Примеры:

- "Запомнил: буду учитывать, что тебе лучше подходят короткие практические задачи."
- "Сохранил это как наблюдение по курсу итальянского."
- "Сохранил контекст на эту дату."
- "Готово, напоминание создано."
- "Готово, дедлайн сохранен."

## Если не уверен

Если есть явный курс — выбирай course observation.
Если есть явная дата/период — выбирай date observation.
Если есть просьба "напомни" — выбирай reminder.
Если есть срок "до/к/дедлайн/экзамен" — выбирай deadline.
Если это устойчивое предпочтение без курса и даты — выбирай user observation.
